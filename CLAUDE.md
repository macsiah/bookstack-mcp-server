## Project Overview

This is a **Model Context Protocol (MCP) server** that exposes the [BookStack](https://www.bookstackapp.com/) knowledge-management API as structured tools and resources consumable by MCP-aware AI clients (e.g. Claude Desktop).

- **Transport:** stdio (not HTTP) — the server is spawned as a subprocess by MCP clients.
- **51 tools** covering every BookStack API endpoint, plus 4 custom tag-management tools.
- **6 resource providers** giving read-only URI-addressable access to content.
- Forked from [pnocera/bookstack-mcp-server](https://github.com/pnocera/bookstack-mcp-server); the 4 `TagTools` in `src/tools/tags.ts` are fork-specific additions developed for **Sunnyside School District** IT operations.

## Architecture

### Core Class: `BookStackMCPServer` (`src/server.ts`)

The server initialises four singletons/shared objects and wires everything together:

```
ConfigManager (singleton)
  └─ BookStackClient (Axios, retry, rate limiter)
  └─ ValidationHandler (Zod schemas)
  └─ Logger (Winston singleton)
  └─ ErrorHandler

BookStackMCPServer
  ├─ setupTools()      → registers all MCPTool instances in a Map<name, MCPTool>
  ├─ setupResources()  → registers all MCPResource instances in a Map<uri, MCPResource>
  └─ setupHandlers()   → wires ListTools / CallTool / ListResources / ReadResource
```

The server runs on **stdio transport** (`StdioServerTransport`). No HTTP port is opened by the MCP layer (express/cors/helmet deps are present but unused by the core server path).

### Singleton Pattern

`ConfigManager`, `Logger` and `ErrorHandler` are singletons. Use `Foo.getInstance()` (or `new ErrorHandler(logger)`) — do **not** construct new instances inside tool classes.

### Dependency Injection

Every tool class receives `(client, validator, logger)` via its constructor. This makes unit testing straightforward: pass mocked instances.

## Environment Variables

Copy `.env.example` to `.env`. All variables are parsed and validated by `ConfigManager`.

| Variable | Default | Description |
|---|---|---|
| `BOOKSTACK_BASE_URL` | `http://localhost:8080/api` | BookStack API base URL |
| `BOOKSTACK_API_TOKEN` | *(required)* | API token (`id:secret` format) |
| `BOOKSTACK_TIMEOUT` | `30000` | HTTP timeout in ms |
| `SERVER_NAME` | `bookstack-mcp-server` | MCP server name |
| `SERVER_VERSION` | `1.0.0` | MCP server version |
| `RATE_LIMIT_REQUESTS_PER_MINUTE` | `60` | Requests allowed per minute |
| `RATE_LIMIT_BURST_LIMIT` | `10` | Burst allowance |
| `VALIDATION_ENABLED` | `true` | Enable parameter validation |
| `VALIDATION_STRICT_MODE` | `false` | Fail (vs warn) on unknown params |
| `LOG_LEVEL` | `info` | `error` / `warn` / `info` / `debug` |
| `LOG_FORMAT` | `pretty` | `pretty` or `json` |
| `NODE_ENV` | `development` | `development` or `production` |
| `DEBUG` | `false` | Enable debug mode |

**Production guards** (`ConfigManager.validateForProduction()`):
- `BOOKSTACK_API_TOKEN` must be set
- `BOOKSTACK_BASE_URL` must not be `localhost`
- `DEBUG` must be `false`
- `LOG_LEVEL` must not be `debug`

Authentication header sent to BookStack: `Authorization: Token <BOOKSTACK_API_TOKEN>`

## TypeScript Conventions

- **Strict mode** is on: no implicit `any`, no implicit returns, no implicit `this`.
- **`noImplicitOverride`** is enabled — use the `override` keyword when overriding.
- **Path aliases** (configured in `tsconfig.json` and must be resolved at runtime via `ts-node`):
  - `@/api/*` → `src/api/*`
  - `@/tools/*` → `src/tools/*`
  - `@/resources/*` → `src/resources/*`
  - `@/utils/*` → `src/utils/*`
  - `@/config/*` → `src/config/*`
  - `@/validation/*` → `src/validation/*`
- All shared types live in **`src/types.ts`** — add new entity types there.
- Tests are excluded from compilation (`tsconfig.json` `exclude` list); Jest uses `ts-jest` directly.

## Adding a New Tool

Follow the pattern established by existing tool classes, especially `src/tools/tags.ts` (the most recently added).

1. **Create `src/tools/<domain>.ts`**

```typescript
import { BookStackClient } from '../api/client';
import { ValidationHandler } from '../validation/validator';
import { Logger } from '../utils/logger';
import { MCPTool } from '../types';

export class MyDomainTools {
  constructor(
    private client: BookStackClient,
    private validator: ValidationHandler,
    private logger: Logger
  ) {}

  getTools(): MCPTool[] {
    return [
      this.createMyTool(),
    ];
  }

  private createMyTool(): MCPTool {
    return {
      name: 'bookstack_my_domain_action',
      description: 'Human-readable description for the AI client.',
      inputSchema: {
        type: 'object',
        properties: {
          id: { type: 'number', description: 'Record ID' },
        },
        required: ['id'],
      },
      handler: async (params: Record<string, unknown>) => {
        this.logger.info('my_domain_action', { params });
        const { id } = this.validator.validateParams(params, ['id']);
        const result = await this.client.someMethod(id as number);
        return JSON.stringify(result, null, 2);
      },
    };
  }
}
```

2. **Register in `src/server.ts`** inside `setupTools()`:

```typescript
import { MyDomainTools } from './tools/my-domain';

// inside setupTools():
const myTools = new MyDomainTools(this.client, this.validator, this.logger);
myTools.getTools().forEach(tool => this.tools.set(tool.name, tool));
```

3. **Add Zod validation schemas** to `src/validation/validator.ts` if you need custom parameter shapes.

4. **Add entity types** to `src/types.ts` if you're working with a new API resource.

## Adding a New Resource

1. **Create `src/resources/<entity>.ts`** returning `MCPResource[]` from `getResources()`.

   - URI format: `bookstack://<entity>` (list) and `bookstack://<entity>/{id}` (single item).
   - `mimeType` is always `'application/json'`.
   - `handler` receives the full URI string and must parse any `{id}` segment itself.

2. **Register in `src/server.ts`** inside `setupResources()`:

```typescript
import { MyEntityResources } from './resources/my-entity';

// inside setupResources():
const myResources = new MyEntityResources(this.client, this.logger);
myResources.getResources().forEach(r => this.resources.set(r.uri, r));
```

## Error Handling

`ErrorHandler` (`src/utils/errors.ts`) converts Axios errors to MCP `McpError`:

| HTTP Status | MCP ErrorCode |
|---|---|
| 400, 422 | `ErrorCode.InvalidParams` |
| 401, 403, 404 | `ErrorCode.InvalidRequest` |
| 429, 5xx | `ErrorCode.InternalError` |

Error details always include `{ type, status, details, url, method }`. Stack traces are only appended when `NODE_ENV=development`.

Zod validation errors are caught by `ValidationHandler` and re-thrown as `McpError` with per-field details.

## Fork-Specific Notes (Sunnyside School District)

- `src/tools/tags.ts` — **Do not remove or rename** these tools during upstream merges:
  - `bookstack_tags_search` — searches by tag using BookStack's `[tag:<name>:<value>]` filter syntax
  - `bookstack_tags_list_all` — paginates all content and enumerates unique tag names + counts
  - `bookstack_tags_audit` — generates tagging coverage reports against an expected tag taxonomy
  - `bookstack_tags_bulk_update` — safe bulk add/set/remove with read-before-write guard

- When syncing from upstream (`pnocera/bookstack-mcp-server`), preserve `src/tools/tags.ts` and its registration in `src/server.ts` (`TagTools` import + `setupTools` call).
